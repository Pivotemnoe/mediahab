# PostHub — UI Reference Pack: начните здесь

## Что это за пакет

Это не готовый frontend и не отдельное техническое задание на весь продукт. Это визуальная и UX-спецификация для реализации frontend PostHub в основном репозитории проекта.

В пакете находятся:

- 4 интерактивных HTML-концепта;
- PNG и PDF-референсы;
- реальный Telegram-пост как пример контента;
- карта экранов и пользовательских сценариев;
- правила дизайн-системы;
- план реализации UI по этапам;
- критерии приёмки;
- стартовые команды для Codex.

## Рабочее название

Рабочее название продукта: **Temichev PostHub**.

Короткое название в интерфейсе: **PostHub**.

Подпись бренда: **by Temichev** — опционально.

Название является рабочим и не должно быть захардкожено в компонентах. Оно должно задаваться через конфигурацию бренда и переменные окружения.

## Какая концепция считается основной

Не нужно выбирать один макет целиком. Финальная система собирается из трёх направлений:

1. **Editorial Studio** — базовый desktop-интерфейс, навигация и контент-студия.
2. **Visual Builder** — конструктор проектов, рубрик, полей, правил и лимитов.
3. **Mobile-first PWA** — диктовка и пошаговый сценарий работы с телефона.

**Command Center** используется позднее для тёмной темы, очереди публикаций, аналитики и операционного режима.

## Как открыть макеты

1. Откройте `index.html` в браузере.
2. Перейдите в каждый из четырёх вариантов.
3. Внутри макета переключайте разделы: landing, dashboard, studio, builder, mobile, billing.
4. PNG используйте для быстрого просмотра и передачи в Codex.
5. HTML используйте как источник компоновки и взаимодействий, но не копируйте его в production как есть.

## Как использовать пакет вместе с основным ТЗ

Распакованную папку нужно положить в основной репозиторий, например:

```text
design/ui-reference/
```

Рекомендуемая структура:

```text
repository-root/
├── AGENTS.md
├── docs/
├── apps/
├── services/
└── design/
    └── ui-reference/
        ├── 00_START_HERE_EN.md
        ├── docs/
        ├── codex/
        ├── *.html
        ├── *.png
        └── assets/
```

После этого Codex получает не ZIP, а весь открытый репозиторий с распакованной папкой.

## Правильный порядок работы

1. Codex читает корневой `AGENTS.md` и основное техническое задание.
2. Затем читает `design/ui-reference/00_START_HERE_EN.md`.
3. Сначала выполняет только UI Phase 00: анализ, карта компонентов, дизайн-токены и план.
4. После подтверждения выполняет UI Phase 01.
5. После каждого этапа показывает скриншоты desktop и mobile, результаты тестов и список изменённых файлов.
6. Следующий этап начинается только после подтверждения владельца продукта.

## К чему стремиться

Главный интерфейс должен ощущаться не как сложная CRM и не как обычный автопостер. Это должна быть понятная контентная студия:

```text
Проект → Рубрика → Диктовка/ввод → AI-редактор → версии площадок → проверка → публикация
```

Пользователь без программиста должен уметь:

- создать проект;
- добавить рубрику;
- собрать поля рубрики;
- задать правила и длину;
- загрузить примеры;
- подключить площадки;
- надиктовать материал;
- получить и отредактировать пост;
- опубликовать его.

## Чего не делать

- Не переносить standalone HTML в production один к одному.
- Не делать четыре разные дизайн-системы.
- Не хардкодить рубрики «Что поесть? Армавир» в общих компонентах.
- Не хардкодить название PostHub.
- Не строить только desktop-версию: диктовка с телефона является основным сценарием.
- Не подключать реальные API платформ до утверждения UX публикации.
- Не считать UI готовым без пустых, загрузочных, ошибочных и мобильных состояний.
